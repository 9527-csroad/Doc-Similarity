# Doc-Similarity 项目潜在问题分析

（与当前实现对齐：v2 书籍指纹 = 伪页 + 前/中/后片段 + 标题；`merged` / `pooled` 双索引；无全书 chunk RAG。）

## 一、相似度计算准确性

### 1.1 问题本质

当前：全文按 `CHARS_PER_PSEUDO_PAGE` 切成伪页；伪页数 ≥8 时取 **前 3 + 中间 3 + 最后 2 + 标题** 共 9 段信息，形成 **merged**（拼接后一次 embedding）与 **pooled**（分段 embedding 再加权平均）两条指纹；伪页数 &lt;8 时用 **全文** 作为单段指纹。检索/去重按请求 `fingerprint_mode` 选用对应 FAISS 索引。

**风险**：

| 场景 | 可能后果 |
|------|----------|
| 同书不同版 | 修订若集中在未采样的伪页间隙 → 两指纹仍可能很接近 |
| 不同书前言/系列雷同 | 前 3 伪页重叠高 → 误判相似 |
| 伪页切分过粗/过细 | 粗则一页信息杂；细则页数膨胀，「中间 3 页」位置敏感 |
| 短文本 | 伪页 &lt;8 时整段嵌入，与长书采样策略不一致，可比性需业务上接受 |

### 1.2 缓解手段

| 手段 | 说明 |
|------|------|
| 查重优先用 hash/ISBN | 已做。content_hash、ISBN 能覆盖绝大部分同书，向量只作补充 |
| 调高向量查重阈值 | 当前 0.9999，已较严，可保持 |
| 多向量取最相似 | 若升级为 chunk 级，可对多 chunk 分别算相似度，取 max 或加权 |
| 明确业务边界 | 相似度用于「推荐/聚类」时可放宽；用于「严格查重」时应依赖 hash/ISBN |

### 1.3 结论

**有风险，但在当前业务下可接受**：查重主要靠 hash/ISBN，向量是兜底；相似推荐不要求 100% 精确。若后续做「严格法律/版权查重」，需考虑多 chunk 或全本分段向量化。

## 二、数据存储与规模

### 2.1 数据库（SQLite / PostgreSQL）

| 表 | 预估规模 | 风险 |
|----|----------|------|
| books | 万级 | 有 `content_hash`、`isbn` 索引，万级无压力 |
| book_uploads | 数万 ~ 十万 | 有 `pdf_id`、`book_id` 索引，十万级可接受 |

**索引**：`content_hash`、`isbn`、`pdf_id`、`book_id` 均已建，查询不会随数据量线性变慢。

**潜在问题**：
- SQLite：单文件、无并发写优化，写入多时可能锁表
- PostgreSQL：万级无压力，十万级需关注连接池、慢查询

### 2.2 向量库（FAISS）

| 项目 | 当前实现 | 规模影响 |
|------|----------|----------|
| 索引类型 | IndexFlatIP（暴力） | 检索 O(n)，与向量数线性相关 |
| 删除 | 全量重建 | 每次 delete 重建整个索引，成本高 |
| 持久化 | 单文件 index + meta.json | 索引变大后加载/保存变慢 |
| 内存 | 全量加载 | 10 万 × 1024 × 4B ≈ 400MB，可接受 |

**量级参考**：

| 向量数 | 单次检索耗时（量级） | 单次 delete 耗时 |
|--------|----------------------|------------------|
| 1 万 | <50ms | 1–3s |
| 10 万 | 100–300ms | 10–30s |
| 50 万 | 0.5–2s | 分钟级 |

### 2.3 结论与应对

| 规模 | 结论 |
|------|------|
| 万级 | 当前方案基本可接受 |
| 10 万+ | 建议切到 IVF/HNSW 等近似索引，或迁移 Milvus |
| 删除频繁 | 需优化：批量删除、延迟重建，或换支持增量删除的向量库 |

## 三、其他潜在问题

### 3.1 数据一致性

- 向量已写入、数据库事务失败 → 产生孤儿向量
- 建议：先写 DB、commit 成功后再写向量；或引入简单补偿/重试

### 3.2 上传链路

- URL 无效、下载超时、txt 编码异常 → 单次失败
- 同步阻塞 5–30s，调用方需设合理超时（如 ≥60s）
- 并发上传时，embedding 和 FAISS 可能成瓶颈

### 3.3 单点与恢复

- FAISS 索引、meta 为本地文件，损坏则需从数据库重新构建
- 建议：定期备份 `index.bin`、`meta.json`，或保留「从 DB 全量重建向量」的脚本

### 3.4 模型与资源

- BGE-M3 加载占内存（约 2–4GB），首请求较慢
- 若部署多实例，每实例各自加载模型，总内存占用会放大

## 四、风险优先级

| 优先级 | 问题 | 触发条件 | 建议 |
|--------|------|----------|------|
| 高 | 向量删除全量重建 | 数据量增大 + 有删除需求 | 提前规划索引升级或换 Milvus |
| 中 | 相似度误判 | 不同书开头相似 | 明确业务边界，必要时多 chunk |
| 中 | DB 与向量不一致 | 写入过程异常 | 调整写入顺序或加补偿逻辑 |
| 低 | SQLite 并发 | 高并发写入 | 换 PostgreSQL 或做写入限流 |
| 低 | 索引文件损坏 | 磁盘/断电 | 备份 + 重建脚本 |

---

## 五、简要结论

1. **相似度**：采样有信息损失，但对当前查重/相似推荐场景可接受；严格查重需多 chunk 或更完整表示。
2. **存储与性能**：万级无大问题；到 10 万+ 需考虑 FAISS 索引类型升级或迁移 Milvus，并优化删除策略。
3. **一致性、上传、恢复**：属常规工程问题，可通过流程和脚本逐步完善。
