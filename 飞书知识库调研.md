# 飞书知识库调研

## 一、整体定位

飞书知识库 = 企业级 AI 知识库 + 知识问答。每个企业有独立知识空间，支持文档、群聊、会议、邮箱等多源数据，用户模糊提问即可获得精准答案。

---

## 二、大体量知识库如何构建

### 2.1 数据来源


| 类型   | 内容                                 |
| ---- | ---------------------------------- |
| 云文档  | 知识库、文档、多维表格、妙记、思维笔记等               |
| 即时消息 | 群聊、私聊（有权限的消息）                      |
| 会议   | 智能纪要、逐字稿                           |
| 邮箱   | 飞书企业邮箱，支持 QQ/腾讯/163/网易等，近 1 年及新增邮件 |
| 表格   | 飞书表格、多维表格                          |


### 2.2 构建流程（技术侧）

```
数据拉取 → 格式转换 → 文本切片 → 向量化 → 向量库
```

- **格式转换**：飞书块级 API 拉取，`BlocksToMarkdown` 转 Markdown，保留标题、代码块、表格等语义
- **切片策略**：按标题、空行等自然分隔符切；纯文本 500–800 tokens，FAQ/代码 200–300 tokens
- **向量库**：支持 Milvus 等向量库，语义检索
- **增量同步**：按修改时间/版本做增量，避免全量重建

### 2.3 体量支撑

- 单表可支持千万级记录（多维表格）
- 权限隔离：仅返回用户有权限的内容
- 多租户：企业级隔离，按租户/空间组织

## 三、精细则如何做到

### 3.1 语义检索（非关键词）

- **向量模型**：基于 BERT 的语义模型（SBERT → 交互增强 → 对比学习 SimCSE）
- **检索方式**：Query 向量化后与知识库向量做相似度（如余弦），而非简单关键词匹配
- **效果**：模糊描述也能命中，如「昨天谁跟我聊了 xx」「哪个群说过 xx」

### 3.2 群聊与对话中的触发


| 能力    | 实现                        |
| ----- | ------------------------- |
| 群聊总结  | 新进群/消息多时，AI 总结历史消息，推荐相关问题 |
| 智能回复  | 收到业务问题时，自动检索知识库并推荐答案      |
| @知识问答 | 群内 @ 提问，答案仅自己可见，可转发       |
| 全局搜索  | 搜索框输入问题，AI 判断是否走知识问答检索    |


### 3.3 精细化的关键点

1. **权限**：检索结果严格按文档/消息权限过滤
2. **溯源**：答案带出处，可跳转原文
3. **上下文**：切片保留结构（标题、段落），便于理解
4. **多模态**：支持图片提问、划词解释
5. **无感融入**：浮窗、划词、@、定时推送等，无需单独打开知识库

## 六、体量估算（600 人企业）

### 6.1 日均数据量（经验估算）


| 类型   | 日均量         | 说明         |
| ---- | ----------- | ---------- |
| 消息   | 3000–6000 条 | 约 5–10 条/人 |
| 文档   | 50–200 个    | 新建/修改      |
| 会议纪要 | 20–50 场     | 视会议频率      |


### 6.2 切片与向量量

- 消息：约 30% 短/无意义被过滤，有效 2000–4000 条，均 0.5–1 chunk → **1000–4000 chunks/天**
- 文档：均 2–5 chunk → **100–1000 chunks/天**
- 会议：均 3–10 chunk → **60–500 chunks/天**

**合计：约 1500–5500 chunks/天**，取中值 **3000 chunks/天**。

### 6.3 存储规模（1024 维 float32）


| 周期  | 向量条数  | 纯向量     | 含索引与元数据（约 2x） |
| --- | ----- | ------- | ------------- |
| 日   | 3k    | ~12 MB  | ~25 MB        |
| 月   | 9 万   | ~360 MB | ~0.7 GB       |
| 年   | 110 万 | ~4.4 GB | ~9 GB         |
| 3 年 | 330 万 | ~13 GB  | ~27 GB        |


公式：条数 × 1024 × 4 字节 × 1.2。若用 IVF_PQ 量化可压至约 1/8。

### 6.4 结论

600 人企业年增约 100 万向量，存储约 **10 GB 级**，可接受。体量主要来自消息。

## 七、无用信息的识别与过滤

### 7.1 不参与向量化的内容


| 类型   | 示例           | 处理        |
| ---- | ------------ | --------- |
| 权限排除 | 用户无权限的文档/群   | 不索引或检索时过滤 |
| 邮箱排除 | 垃圾箱、屏蔽邮件     | 拉取时排除     |
| 时间范围 | 邮箱仅近 1 年     | 按策略限定     |
| 短消息  | 「收到」「好的」「ok」 | 长度/规则过滤   |
| 系统消息 | 入群提醒、@ 通知    | 类型过滤      |
| 纯表情  | 无文字          | 跳过        |


### 7.2 识别手段


| 手段   | 适用                    |
| ---- | --------------------- |
| 规则   | 长度 < N 字、停用词、正则       |
| 类型标记 | 按 message_type 排除系统消息 |
| 分类模型 | BERT 二分类：有效/无效        |
| 语义去重 | 相似度 > 0.95 只保留一条      |
| 质量打分 | 错别字率、信息密度，低于阈值不索引     |


### 7.3 业界经验

企业知识库噪声占比可达 **30%+**。聊天场景「收到」「好的」多，有效密度低，需前置过滤。飞书检索按权限过滤，文档可配置搜索范围。
